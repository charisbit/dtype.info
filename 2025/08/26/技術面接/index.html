私わたしは中国ちゅうごくの名門めいもん大学だいがくで電気でんき工学こうがくとシステムしすてむ分析ぶんせきを学んだまなんだ後あと、来らい日にちしてから八はち年ねん以上いじょうにわたり、完全かんぜん日本にほん語ご環境かんきょうでソフトウェアそふとうぇあ開発かいはつに従事じゅうじしてきました。これまでの経験けいけんでは、ウェブシステムしすてむから機械きかい学習がくしゅう基盤きばん、そして最近さいきんでは生成きなりAIを活用かつようした運用うんよう監視かんし自動じどう化かシステムしすてむや社しゃ内ないエージェントシステムしすてむの構築こうちくまで、多岐にたきにわたるプロジェクトぷろじぇくとを主導しゅどう・開発かいはつしてきました。

スライドすらいど1: プロジェクトぷろじぇくと概要がいよう

Intelligent Presenter for HTML Slidesは、BacklogプロジェクトぷろじぇくとデータでーたからAIを活用かつようしてHTMLスライドすらいどを自動じどう生成せいせいするシステムしすてむです。

主要しゅよう機能きのう

  • BacklogプロジェクトぷろじぇくとデータでーたをMCP協議きょうぎで取得しゅとく
  • AIによる10種類しゅるいのテーマてーま別べつスライドすらいど自動じどう生成せいせい
  • 日本にほん語ごTTS音声おんせい解説かいせつ機能きのう
  • リアルりあるタイムたいむ流りゅう式しき処理しょりによるスライドすらいど配信はいしん

解説かいせつ文ぶん

こんにちは。私わたしが開発かいはつしたIntelligent Presenter for HTML Slidesについてご紹介しょうかいさせていただきます。このシステムしすてむは、BacklogのプロジェクトぷろじぇくとデータでーたからAIを使ってつかってHTMLスライドすらいどを自動じどうで生成せいせいするシステムしすてむです。Backlogプロジェクトぷろじぇくとのデータでーたを自動じどうで取得しゅとくし、AIが十じゅう種類しゅるいのテーマてーまに分けてわけてスライドすらいどを作成さくせいします。さらに、日本にほん語ごの音声おんせい解説かいせつも付けるつけることができます。

スライドすらいど2: 技術ぎじゅつアーキテクチャ

主要しゅようコンポーネント

  • フロントエンド: Vue 3 + TypeScript + Slidev
  • バックばっくエンドえんど: Go + Gin + WebSocket
  • MCP統合とうごう: Backlog MCP Server + Speech TTS Server
  • 認証にんしょう: OAuth 2.0 + JWT

解説かいせつ文ぶん

システムしすてむのアーキテクチャは三さん層そう構造こうぞうになっています。フロントエンドはVue 3とTypeScript、Slidevを使用しようしています。バックエンドはGo言語げんごとGinフレームワークで構築こうちくし、WebSocketでリアルりあるタイムたいむ通信つうしんを行いおこないます。外部がいぶサービスさーびすとの連携れんけいはMCP協議きょうぎを使いつかい、Backlog MCP ServerとTTS音声おんせい合成ごうせいサーバーさーばーに接続せつぞくしています。認証にんしょうはOAuth 2.0とJWTを使用しようして実装じっそうしました。

プロジェクトぷろじぇくと管理かんりの全ぜん側面そくめんをカバーかばー

  1. プロジェクトぷろじぇくと概況がいきょうと基本きほん情報じょうほう
  2. プロジェクトぷろじぇくと進度しんどと状態じょうたい
  3. 課題かだい詳細しょうさいと管理かんり
  4. プロジェクトぷろじぇくとリスクりすくと瓶頸予警
  5. チームちーむメンバーめんばーと協作状態じょうたい
  6. ドキュメントどきゅめんとと知識ちしき庫こ管理かんり
  7. コードこーどベースべーすと開発かいはつ活動かつどう
  8. 通知つうちとコミュニケーションこみゅにけーしょん管理かんり
  9. 智能化か輔助と予測よそく分析ぶんせき
  10. 総括そうかつと次つぎ段階だんかい計画けいかく

解説かいせつ文ぶん

システムしすてむでは十じゅう種類しゅるいのスライドすらいどテーマてーまを用意よういしています。プロジェクトぷろじぇくと概要がいようから始まりはじまり、進捗しんちょく状況じょうきょう、課題かだい管理かんり、リスクりすく分析ぶんせき、チームちーむ協力きょうりょく状況じょうきょう、文書ぶんしょ管理かんり、開発かいはつ活動かつどう、コミュニケーションこみゅにけーしょん管理かんり、そして予測よそく分析ぶんせきまで、プロジェクトぷろじぇくと管理かんりの全てすべての側面そくめんをカバーかばーしています。最後にさいごに総括そうかつと次のつぎの段階だんかいの計画けいかくを自動じどう生成せいせいします。これにより、包括ほうかつ的なてきなプロジェクトぷろじぇくとレポートれぽーとを短たん時間じかんで作成さくせいできます。

スライドすらいど4: リアルりあるタイムたいむ生成せいせいフローふろー

非ひ同期どうき流りゅう式しき処理しょり

graph TD
    A[ユーザー要求] --> B[Backlogデータ取得]
    B --> C[AI内容生成]
    C --> D[スライド配信]
    D --> E[TTS音声生成]
    E --> F[完了通知]
    
    B -.-> G[WebSocket]
    C -.-> G
    D -.-> G
    E -.-> G

解説かいせつ文ぶん

システムしすてむの生成せいせいフローふろーは非ひ同期どうきで処理しょりされます。ユーザーゆーざーからのリクエストりくえすとを受けるうけると、まずBacklogからプロジェクトぷろじぇくとデータでーたを取得しゅとくします。次につぎに、AIがそのデータでーたを分析ぶんせきしてスライドすらいどの内容ないようを生成せいせいし、WebSocketを通じてつうじてリアルりあるタイムたいむでフロントエンドに配信はいしんします。同時にどうじに、TTS機能きのうで日本にほん語ごの音声おんせい解説かいせつも生成せいせいします。すべての処理しょりが完了かんりょうすると、完了かんりょう通知つうちをユーザーゆーざーに送信そうしんします。

システムしすてむは複数ふくすうのコンテンツこんてんつ表現ひょうげん方法ほうほうを統合とうごうしています。MarkdownとMermaidでフローチャートや図表ずひょうを自動じどう生成せいせいし、Chart.jsでインタラクティブなデータでーた可視化かを実現じつげんします。Slidevを使ってつかってモダンなもだんなHTML5プレゼンテーションを作成さくせいし、TTS機能きのうで日本にほん語ごのナレーションなれーしょんも自動じどう生成せいせいします。AIの統合とうごうでは、OpenAI GPT-4とAWS Bedrockを使用しようし、RealtimeTTSで高こう品質ひんしつな日本にほん語ご音声おんせいを生成せいせいします。

特にとくに最新のさいしんのプロジェクトぷろじぇくとでは、AWSの生成せいせいAIサンプルさんぷるプロジェクトぷろじぇくとを活用かつようした社しゃ内ないエージェントシステムしすてむ「Intelligent Agent Platform」の設計せっけい・実装じっそうを担当たんとうし、MCP（Model Context Protocol）を採用さいようしたサーさーバーレス構成こうせいによるスケーラブルな自動じどう化かシステムしすてむを構築こうちくしました。また、複数ふくすうのLLMモデルもでる（Claude 3 Haiku、Cohere Command R+など）の統合とうごうや、RAG機能きのうを使ったつかった知識ちしき検索けんさくシステムしすてむの実装じっそうも経験けいけんしています。

技術ぎじゅつスタックとしては、バックエンドではGo、Python、Scala、フロントふろんとエンドえんどではTypeScript、React、Vue.jsを使用しようし、クラウドではAWS、Azure、GCPすべてでの開発かいはつ・運用うんよう経験けいけんがあります。特にとくにAI・機械きかい学習がくしゅう分野ぶんやでは、LangChain、TensorFlow、PyTorchを使ったつかった実装じっそう経験けいけんを持ちもち、最近さいきんはOpenAI API、AWS Bedrock、Anthropic Claudeなどの最新さいしんLLMサービスさーびすの統合とうごう開発かいはつに深くふかく関わってかかわっています。

今回こんかいの技術ぎじゅつ課題かだいでは、「Intelligent Presenter for HTML Slides」というプロダクトを開発かいはつし、BacklogのプロジェクトぷろじぇくとデータでーたからAIを活用かつようしてHTMLスライドすらいどを自動じどう生成せいせいするシステムしすてむを構築こうちくしました。MCP協議きょうぎの採用さいよう、Vue 3とSlidevによるモダンなもだんなフロントエンド、そして日本にほん語ごTTS機能きのうまで統合とうごうした包括ほうかつ的なてきなソリューションそりゅーしょんに仕上げしあげました。この課題かだいを通じてつうじて、Nulabの技術ぎじゅつ方針ほうしんやプロダクト理念りねんに深くふかく共感きょうかんし、特にとくにBacklog AI助手じょしゅの機能きのう拡張かくちょうという方向ほうこう性せいで、既存きそんサービスさーびすとの統合とうごうを意識いしきした設計せっけいを心がけこころがけました。

私わたしがNulabに魅力みりょくを感じるかんじるのは、「このチームちーむで一緒にいっしょに仕事しごとできてよかった」という理念りねんと、TRY FIRST、LOVE DIFFERENCES、GOAL ORIENTEDという行動こうどう方針ほうしんです。これまでの経験けいけんでも、多様なたようなメンバーめんばーとのチームワークを重視じゅうしし、新しいあたらしい技術ぎじゅつへの挑戦ちょうせんを恐れおそれず、常につねに共通のきょうつうの目標もくひょうに向かってむかって努力どりょくすることを大切にたいせつにしてきました。

Nulabでは、これまでの豊富なほうふな開発かいはつ経験けいけんとAI技術ぎじゅつへの深いふかい理解りかいを活かしいかし、Backlog AI助手じょしゅをはじめとする次つぎ世代せだいプロダクトの開発かいはつに貢献こうけんしたいと考えてかんがえています。特にとくに、ユーザーゆーざーの業務ぎょうむ効率こうりつ化かと新しいあたらしい価値かち創造そうぞうの両立りょうりつを目指しめざし、技術ぎじゅつ的なてきな革新かくしんとユーザビリティの向上こうじょうを同時にどうじに実現じつげんするプロダクト開発かいはつに取り組みとりくみたいと思いおもいます。

面接Q&A集（日本語）

面接めんせつ官かんからの予想よそう質問しつもんと回答かいとう

Q1: 今回こんかいの技術ぎじゅつ課題かだいでなぜこのアーキテクチャを選択せんたくしましたか？

A1:
三さんつの主要なしゅような理由りゆうがあります。まず、Nulabの技術ぎじゅつスタックとの親和しんわ性せいを重視じゅうしし、TypeScriptとGoという御おん社しゃが推奨すいしょうする技術ぎじゅつを採用さいようしました。次につぎに、スケーラビリティを考慮こうりょし、フロントふろんとエンドえんどとバックエンドを分離ぶんりしたマイクロサービスさーびす指向しこうのアーキテクチャを設計せっけいしました。最後にさいごに、将来しょうらい的なてきな機能きのう拡張かくちょうを見据えてみすえて、MCP協議きょうぎによる柔軟なじゅうなんなサービスさーびす統合とうごうを実現じつげんしました。これにより、新しいあたらしいデータでーたソースそーすやAIサービスさーびすを容易によういに追加ついかできる拡張かくちょう性せいの高いたかいシステムしすてむになっています。

Q2: OAuth 2.0を選択せんたくした理由りゆうと実装じっそうで工夫くふうした点てんを教えておしえてください。

A2:
OAuth 2.0の選択せんたくは、セキュリティせきゅりてぃ要件ようけんとユーザビリティの両立りょうりつを目的もくてきとしています。APIキーきー認証にんしょうと比べてくらべて、ユーザーゆーざーのクレデンシャル情報じょうほうを直接ちょくせつ扱わあつかわず、スコープ制限せいげんにより必要ひつよう最小さいしょう限げんの権限けんげんのみアクセスあくせすできるため、セキュリティリスクを大幅におおはばに軽減けいげんできます。実装じっそうでは、CSRF攻撃こうげきを防ぐふせぐためのstate parameterによる検証けんしょう、JWT tokenの適切なてきせつな有効ゆうこう期限きげん設定せってい、そして認証にんしょう状態じょうたいの効率こうりつ的なてきなキャッシュきゃっしゅ管理かんりを実装じっそうしました。また、refresh tokenによる自動じどう更新こうしん機能きのうも組みくみ込みこみ、ユーザーゆーざーの継続けいぞく的なてきな利用りよう体験たいけんを確保かくほしています。

Q3: MCP（Model Context Protocol）を採用さいようした技術ぎじゅつ的なてきな理由りゆうは何なにですか？

A3:
MCP採用さいようの主なおもな理由りゆうは三みっつあります。第だい一いちに、標準ひょうじゅん化かされたプロトコルを使用しようすることで、異なることなるサービスさーびす間かんの統合とうごうが容易によういになります。第だい二にに、JSON-RPC 2.0ベースべーすの非ひ同期どうき通信つうしんにより、リアルりあるタイムたいむ処理しょりとパフォーマンスぱふぉーまんすの最適さいてき化かを実現じつげんできます。第だい三さんに、Nulabが公式にこうしきに提供ていきょうするbacklog-mcp-serverを活用かつようすることで、開発かいはつ効率こうりつと保守ほしゅ性せいを向上こうじょうできます。また、将来しょうらい的にてきにClaude DesktopやCursorなどの外部がいぶAIツールつーるとの統合とうごうも容易によういになるため、エコえこシステムしすてむの拡張かくちょう性せいも考慮こうりょしました。

Q4: リアルりあるタイムたいむ処理しょりでWebSocketを選択せんたくした理由りゆうを説明せつめいしてください。

A4:
WebSocketを選択せんたくした理由りゆうは、スライドすらいど生成のきなりの非ひ同期どうき処理しょりにおける優れたすぐれたユーザーゆーざー体験たいけんの実現じつげんです。従来じゅうらいのHTTPポーリングと比べてくらべて、WebSocketは双方向そうほうこう通信つうしんにより、サーバーさーばー側がわからクライアントくらいあんとへリアルりあるタイムたいむで進捗しんちょく情報じょうほうを配信はいしんできます。スライドすらいど生成せいせいは時間じかんのかかる処理しょりのため、ユーザーゆーざーがリアルりあるタイムたいむで進行しんこう状況じょうきょうを確認かくにんできることが重要ですじゅうようです。また、ネットワークねっとわーく効率こうりつの観点かんてんでも、持続じぞく的なてきな接続せつぞくによりオーバーおーばーヘッドへっどを削減さくげんし、複数ふくすうのスライドすらいどデータでーたを段階だんかい的にてきに配信はいしんする際さいの性能せいのう向上こうじょうを実現じつげんできます。

Q5: AIの統合とうごうにおいて、OpenAIとAWS Bedrockの両方りょうほうを実装じっそうした理由りゆうは？

A5:
複数ふくすうのAIプロバイダーぷろばいだー統合とうごうは、可用性かようせいとリスクりすく分散ぶんさんを目的もくてきとしています。OpenAI APIは高こう品質ひんしつな生成せいせい能力のうりょくを持ちもちますが、サービスさーびす停止ていしやレートれーと制限せいげんのリスクりすくがあります。AWS Bedrockは複数ふくすうのモデルもでる（Claude、Command、Jurassic）へのアクセスあくせすを提供ていきょうし、企業きぎょうレベルれべるのSLAと地域ちいき対応たいおうを実現じつげんします。フォールバック機能きのうにより、一ひとつのサービスさーびすに問題もんだいが発生はっせいした場合ばあいでも継続けいぞく的なてきな利用りようが可能ですかのうです。また、コストこすと最適さいてき化かの観点かんてんでも、用途ようとに応じておうじて最適なさいてきなモデルもでるを選択せんたくすることで、運用うんようコストこすとの削減さくげんを図るはかることができます。

Q6: テストてすと戦略せんりゃくについて説明せつめいしてください。どのようなテストてすとを実装じっそうしましたか？

A6:
包括ほうかつ的なてきなテストてすと戦略せんりゃくを実装じっそうしました。Go backend では単体たんたいテストてすとで80%のコードカバレッジを達成たっせいし、特にとくに認証にんしょうロジック、MCP統合とうごう、WebSocket通信つうしんの重要なじゅうような部分ぶぶんは100%カバーかばーしています。TypeScript frontend では、Vitestを使用しようしたコンポーネントテストてすとと統合とうごうテストてすとを実装じっそうし、APIとの連携れんけい部分ぶぶんやスライドすらいど生成せいせいフローふろーの検証けんしょうを行っておこなっています。また、Docker環境かんきょうでの結合けつごうテストてすとにより、実際じっさいのデプロイ環境かんきょうでの動作どうさ確認かくにんも実施じっししています。テストてすとの自動じどう化かにより、CI/CDパイプぱいぷラインらいんでの品質ひんしつ保証ほしょうも実現じつげんしています。

Q7: パフォーマンスぱふぉーまんす最適さいてき化かでどのような工夫くふうをしましたか？

A7:
複数ふくすうの段階だんかいでパフォーマンスぱふぉーまんす最適さいてき化かを実施じっししました。バックエンドでは、Go routineによる並行へいこう処理しょりでBacklogデータでーた取得しゅとくとAI生成せいせいを同時どうじ実行じっこうし、処理しょり時間じかんを短縮たんしゅくしています。フロントエンドでは、Vue 3のComposition APIとreactivity systemを活用かつようし、不要なふような再さい描画びょうがを最小さいしょう限げんに抑制よくせいしています。また、WebSocketによるストリーミング配信はいしんで、スライドすらいどを生成せいせい次第しだい順次じゅんじ配信はいしんすることで、ユーザーゆーざーの待機たいき時間じかんを短縮たんしゅくしています。Redisによるキャッシングで、同じおなじプロジェクトぷろじぇくとの繰り返しくりかえしアクセスあくせす時じの応答おうとう速度そくども向上こうじょうさせています。

Q8: セキュリティせきゅりてぃ面めんでどのような対策たいさくを実装じっそうしていますか？

A8:
多層たそう的なてきなセキュリティせきゅりてぃ対策たいさくを実装じっそうしています。認証にんしょうレベルれべるでは、OAuth 2.0によるセキュアな認証にんしょうフローふろー、JWTトークンの適切なてきせつな有効ゆうこう期限きげん管理かんり、CSRF対策たいさくのためのstateパラぱらメーターめーたー検証けんしょうを実装じっそうしています。通信つうしんレベルれべるでは、HTTPS強制きょうせい、WebSocketのSecure接続せつぞく、CORS設定せっていによる適切なてきせつなオリジン制限せいげんを行っておこなっています。データでーた保護ほごでは、機密きみつ情報じょうほうの環境かんきょう変数へんすう管理かんり、APIキーきーの安全なあんぜんな保存ほぞん、ログろぐでの個人こじん情報じょうほうマスキングを実施じっししています。また、Docker securityの観点かんてんでも、non-rootユーザーゆーざーでの実行じっこう、最小さいしょう権限けんげんの原則げんそくを適用てきようしています。
